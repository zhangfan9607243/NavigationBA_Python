# Topic 3.1 - Python 面向对象的程序开发概览

## 1. Python 面向过程的程序开发回顾

在 Python 基础模块中，我们介绍了 Python 程序开发的一般流程和规范，面向对象的程序开发也基本遵循上一模块中介绍到的流程和规范，在此基础上只有很少的一些补充内容需要注意。

我们首先来简单复习一下上一模块中介绍到的 Python 程序开发的一般流程：

- (1) **需求分析**：明确程序需要实现的功能和目标用户
- (2) **设计**：根据需求设计程序的整体架构，包括模块划分、数据结构选择等
- (3) **编码**：根据设计文档编写代码，注意代码的可读性和可维护性
- (4) **测试**：对程序进行全面测试，包括单元测试、集成测试等，确保程序的正确性和稳定性
- (5) **部署**：将程序部署到生产环境，供用户使用
- (6) **维护**：根据用户反馈和需求变化，对程序进行迭代和优化

在这个流程指导下，一种常见的 Python 项目文件结构如下：

```text
project_name/
│
├── data/                 ← 数据文件夹：原始数据、输入输出文件
│   ├── data1.txt
│   ├── data2.csv
│   └── data3.json
│
├── src/                  ← 源代码（Source Code）核心程序逻辑
│   ├── __init__.py       ← 标识这是一个包（可根据需要添加）   
│   ├── main.py           ← 主程序入口
│   ├── config.py         ← 配置参数与路径
│   ├── data_loader.py    ← 数据读取与保存
│   ├── analysis.py       ← 分析或建模逻辑
│   ├── report.py         ← 报告与输出
│   └── utils.py          ← 工具函数、日志、文件操作
│
├── tests/                ← 测试代码（确保程序正确性）
│   ├── __init__.py       ← 标识这是一个包（可根据需要添加）   
│   ├── test_main.py
│   ├── test_data_loader.py
│   ├── test_analysis.py
│   ├── test_report.py
│   └── test_utils.py
│
├── logs/                 ← 程序运行日志
│   └── app.log
│
├── requirements.txt      ← 依赖清单（环境管理）
└── README.md             ← 项目说明文档
```

这个文件目录结构并不是固定的，可以根据具体项目的需求进行调整，但总体上应遵循模块化、层次化的原则，确保代码的可维护性和可扩展性。

## 2. 面向对象程序开发的补充注意事项

### (1) 面向对象程序开发的流程

Python 面向对象程序开发同样遵循上述的流程，但在设计阶段有一些补充的注意事项：

1. **类的设计**：

    - 在设计时考虑类的职责划分，设计每个类的属性和方法

    - 确保每个类有**明确且单一**的职责，避免一个类承担过多功能

    - 例如植物大战僵尸这个游戏中：
    
        - 有地图类、植物类、僵尸类等，每个类负责不同的功能，不存在哪个类承担过多职责的情况
        - 具体来说，植物类的属性有生命值、攻击力、攻击范围等，方法有攻击、被攻击等


2. **类之间的关系**：

    - 考虑类之间的关系，如继承、关联和组合等

    - 要站在宏观的角度设计类之间的交互，类之间关系梳理明白了，整个程序的运行逻辑就出来了

    - 例如在植物大战僵尸中：
    
        - 继承关系包括：向日葵类、豌豆射手类、坚果类等都继承自植物类
        - 关联关系包括：地图对象上面有植物对象和僵尸对象
        - 组合关系包括：植物类在执行攻击方法时，子弹对象才会被创建
        - 当我们把所有类之间的关系梳理清楚后，整个游戏的运行逻辑就出来了

3. **特殊的类**：

    - 考虑是否有必要安排一些特殊的类，如抽象类、工具类、配置类、单例类等

    - 例如在植物大战僵尸中：
    
        - 植物类就可以设计成一个抽象类，定义一些通用的属性和方法，然后具体的植物类继承自植物类并实现具体的方法
        - 游戏中的工具类可以包含一些通用的功能函数，如计算距离、计算伤害、生成随机数等
        - 游戏所有的配置参数可以放在一个配置类中，方便管理和访问
        - 游戏中的地图就可以设计成单例类，确保整个游戏过程中只有一个地图对象存在

### (2) 面向对象程序开发的项目组织形式

在面向对象程序开发中，项目的文件组织形式和上一模块介绍的基本相同，只有一个额外需要注意的点，就是类放在哪个文件里。

通常情况下：

- 类的定义都放在 `src/` 目录下的单独模块文件中，并且每个模块文件通常只定义一个类
- 类之间如果有关系（包括继承、组合、以及关联），则只需要在模块文件中通过 `import` 语句导入相关类即可
- 同级别的类通常放在同一个包（文件夹）下，不同级别的类可以放在不同的包中，例如所有的植物类可以放在 `plants/` 包下，所有的僵尸类可以放在 `zombies/` 包下
- 同样的，类的测试代码也放在 `tests/` 目录下的单独模块文件中，每个测试模块文件通常只测试一个类

除此之外还需要注意命名规范：

- 模块名通常使用小写字母和下划线命名法（snake_case），例如 `plant.py`、`zombie.py`、`pea_shooter.py` 等
- 而类名通常使用打驼峰命名法（CamelCase），也就是首字母大写，例如 `Plant`、`Zombie`、`PeaShooter` 等
- 如果严格按照 Python 推荐的命名规范，文件名和类名应使用不同的命名方法，但是大家可以看到大部分开发者可能觉得麻烦，就直接把文件名和类名都用打驼峰命名法了，例如 植物类名是 `Plant`，文件名是 `Plant.py`，这样也没有什么问题，只要开发时注意区分即可

我们来看一个简单的植物大战僵尸项目的文件组织形式示例

- 虽然网上有许多《植物大战僵尸》重写版（官方游戏并没有开源），但这里我们并没有参考网上代码，因为完整代码太大了，这里我们根据对游戏的理解，设计了一个简单的文件组织形式示例，供大家参考，原版游戏的很多元素，如商店、花园、多种游戏模式等，我们这里并没有体现
